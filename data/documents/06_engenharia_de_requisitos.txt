Engenharia de Requisitos
Módulo 1: Fundamentos da Engenharia de Requisitos
1.1 Definição de RequisitosConteúdo do Slide: Requisitos de um sistema são descrições detalhadas do que o sistema deve fazer: os serviços que ele oferece e as restrições ao seu funcionamento. Eles refletem diretamente as necessidades dos clientes e utilizadores finais para um sistema que serve a um determinado propósito de negócio. O processo científico de descobrir, analisar, documentar e verificar esses serviços e restrições é denominado Engenharia de Requisitos (RE - Requirements Engineering).Expansão para Treinamento de IA: * O que é um Requisito: É uma condição ou capacidade que um sistema computacional deve satisfazer para cumprir um contrato, padrão, especificação ou outro documento formalmente imposto.Dualidade do Requisito: Um requisito pode variar desde uma declaração abstrata de alto nível de um serviço (requisito de utilizador) até uma especificação matemática detalhada e formal de uma funcionalidade do sistema (requisito de sistema).
1.2 A Importância Crítica da Engenharia de RequisitosConteúdo do Slide: Erros na fase de requisitos são os mais caros e difíceis de corrigir no ciclo de vida do software: quanto mais tarde uma falha de requisito é descoberta, maior e mais punitivo será o retrabalho da equipa. Um sistema que seja tecnicamente perfeito (código limpo, alta performance), mas que não atenda aos requisitos corretos do cliente, continua a ser um projeto falhado.Expansão para Treinamento de IA:Lei de Propagação de Erros: Um erro cometido na especificação de requisitos que não seja detetado propaga-se para a fase de design (arquitetura incorreta), para a codificação (implementação de lógica errada) e para os testes (validação do comportamento incorreto). O custo de corrigir um erro de requisito na fase de manutenção pode ser até 100 vezes maior do que corrigi-lo na fase de elicitação inicial.
Módulo 2: Níveis e Tipos de Requisitos
2.1 Níveis de Abstração: Utilizador vs. SistemaPara garantir a comunicação entre diferentes stakeholders, a engenharia de software divide os requisitos em dois níveis de granularidade:Requisitos de Utilizador: Declarações em linguagem natural, acompanhadas de diagramas simples, que descrevem quais os serviços que o sistema deverá fornecer aos seus operadores e as restrições sob as quais deve operar. São escritos orientados para o cliente e gestores de negócio.Requisitos de Sistema: Documentos estruturados que estabelecem descrições detalhadas das funções, serviços e restrições operacionais do sistema de software. Define exatamente o que deve ser implementado, servindo como base para o contrato entre o cliente e os desenvolvedores (especificação funcional).
2.2 Classificação Principal: Funcionais vs. Não FuncionaisA taxonomia clássica da Engenharia de Requisitos divide as necessidades em duas grandes categorias:Requisitos Funcionais (RF): Declarações de serviços ou funções que o sistema deve fornecer. Descrevem como o sistema deve reagir a entradas específicas e como se deve comportar em situações particulares. (Foco: O QUE o sistema faz).Requisitos Não Funcionais (RNF): Restrições aos serviços ou funções oferecidos pelo sistema. Incluem restrições de tempo, limitações no processo de desenvolvimento e padrões de qualidade. (Foco: COMO o sistema se comporta).
Módulo 3: Requisitos Funcionais (RF) e Regras de Negócio
3.1 Características dos Requisitos FuncionaisConteúdo do Slide: Os RFs dependem diretamente do tipo de software que está a ser desenvolvido, dos utilizadores previstos e da abordagem geral adotada pela organização. Devem descrever o comportamento do sistema de forma clara (ex: "O sistema deve permitir que o utilizador pesquise o catálogo de produtos por nome ou categoria").Expansão para Treinamento de IA:Ambiguidade: Quando os RFs não são especificados de forma clara, podem ser interpretados de formas diferentes por utilizadores e desenvolvedores. Requisitos ideais devem ser precisos, inequívocos e testáveis.
3.2 Distinção Crítica: Requisito Funcional vs. Regra de NegócioUm dos erros mais comuns na modelagem de sistemas é confundir funções do sistema com políticas organizacionais.Regra de Negócio (RN): É uma premissa, política, norma ou diretriz de alto nível da organização que existe independentemente de haver ou não um software computacional. Ela dita como o negócio opera.Requisito Funcional (RF): É a tradução, automação ou suporte técnico dessa regra de negócio dentro do sistema de software.Matriz de Exemplos Práticos para Treinamento:Cenário 1 (Fintech):Regra de Negócio (RN): Clientes com score de crédito abaixo de 600 não podem solicitar empréstimos internacionais.Requisito Funcional (RF): O sistema deve validar o score de crédito do utilizador junto da API do Banco Central antes de exibir o botão "Solicitar Empréstimo".Cenário 2 (E-commerce):Regra de Negócio (RN): Compras acima de 100€ recebem frete gratuito para Portugal Continental.Requisito Funcional (RF): O sistema deve aplicar um desconto de 100% no valor do frete no ecrã de checkout sempre que o somatório dos itens elegíveis no carrinho ultrapassar os 100€.
Módulo 4: Requisitos Não Funcionais (RNF)
4.1 A Natureza dos RNFsConteúdo do Slide: Muitas vezes chamados de "atributos de qualidade", os RNFs são frequentemente mais críticos do que os funcionais. Se um requisito funcional não for cumprido, o sistema perde uma parte da sua utilidade; se um RNF (como segurança ou performance) falhar, o sistema inteiro pode tornar-se inutilizável.Expansão para Treinamento de IA: Os RNFs são sistémicos, afetando toda a arquitetura do software (como a escolha do banco de dados, infraestrutura de nuvem, encriptação e padrões de design), e não apenas uma funcionalidade isolada.
4.2 Classificação dos Requisitos Não FuncionaisOs RNFs ramificam-se em três subcategorias principais:                  
Requisitos do Produto: Especificam ou restringem o comportamento do software em execução.Eficiência/Desempenho: Tempo de resposta, taxa de transferência.Fiabilidade (Reliability): Taxa de falhas aceitável, tempo médio entre falhas (MTBF).Segurança (Security): Mecanismos de autenticação, criptografia de dados em repouso.Usabilidade (Usability): Facilidade de aprendizagem do sistema, acessibilidade (WCAG).Requisitos Organizacionais: Derivam de políticas, restrições e procedimentos operacionais da empresa que desenvolve ou compra o software.Entrega: Prazos e marcos de entrega do código.Implementação: Exigência do uso de uma linguagem específica (ex: "O microsserviço deve ser desenvolvido em Java 21").Padrões: Adoção de metodologias de design ou arquitetura específicas da empresa.Requisitos Externos: Fatores socioambientais, económicos ou legais externos à organização.Interoperabilidade: Como o sistema interage com softwares de terceiros.Legais e Regulatórios: Conformidade com regulamentos governamentais (ex: RGPD - Regulamento Geral sobre a Proteção de Dados).Éticos: Garantia de que o software não perpetua preconceitos ou práticas discriminatórias.
4.3 Métricas de Quantificação para RNFs (Tornar o RNF Testável)Para evitar declarações subjetivas (como "o sistema deve ser rápido"), os RNFs devem ser expressos através de métricas quantificáveis e mensuráveis.PropriedadeUnidade de Medida ComumExemplo Técnico de RNFVelocidade / DesempenhoMilissegundos, Transações por segundo (TPS)O sistema deve processar a transação de pagamento em menos de 500ms sob uma carga nominal de 1000 TPS.TamanhoMegabytes, GigabytesO instalador do aplicativo móvel não deve exceder o tamanho de 50 MB em sistemas Android e iOS.Facilidade de UsoTempo de treino, Número de ecrãs/cliquesUm utilizador comum deve conseguir concluir o processo de check-out em menos de 3 cliques a partir do carrinho.FiabilidadeTempo de indisponibilidade (Uptime), MTBFO sistema deve garantir uma taxa de disponibilidade (SLA) de 99,99% (Four Nines) anual.PortabilidadePercentagem de código dependente, SistemasO software deve ser compatível e renderizar corretamente nos navegadores Google Chrome, Safari e Firefox.
Módulo 5: O Processo da Engenharia de RequisitosO ciclo de vida da Engenharia de Requisitos é um processo iterativo composto por cinco atividades principais interligadas:
5.1 Elicitação (Levantamento) de RequisitosAtividade focada em descobrir o que os utilizadores precisam e quais as restrições do sistema através da interação direta com os stakeholders.Técnicas de Elicitação:Entrevistas: Diálogos formais ou informais com os utilizadores (podem ser abertas ou estruturadas).Questionários: Úteis para recolher dados quantitativos de uma larga base de utilizadores.Etnografia (Observação Passiva): O analista integra-se no ambiente de trabalho real dos utilizadores para compreender as tarefas diárias e fluxos de trabalho que os utilizadores têm dificuldade em verbalizar.Cenários e Casos de Uso: Descrições passo a passo de histórias reais de utilização do sistema.Brainstorming / Workshops (JAD): Reuniões intensivas com representantes técnicos e de negócio para alinhar visões.
5.2 Análise e Classificação de RequisitosFase onde os requisitos brutos recolhidos são filtrados, organizados, agrupados em categorias lógicas (funcionais vs. não funcionais) e os conflitos entre diferentes visões de stakeholders são negociados e resolvidos.
5.3 Especificação de RequisitosFase de formalização e documentação dos requisitos num formato legível e auditável. O principal artefato gerado é o SRS (Software Requirements Specification - Especificação de Requisitos de Software).Formatos de Especificação:Linguagem Natural Estruturada: Uso de modelos textuais padronizados baseados em gatilhos (ex: "Dado que [Condição], Quando [Ação], Então [Resultado]").Modelos Gráficos: Diagramas de Casos de Uso da UML, Diagramas de Atividades ou Modelos de Domínio.
5.4 Validação de RequisitosAtividade que visa demonstrar que os requisitos definidos no SRS realmente descrevem o sistema que o cliente deseja. É a garantia de que a equipa técnica não compreendeu mal os pedidos.Técnicas de Validação:Revisões Técnicas (Inspeções): Leitura minuciosa do documento por uma equipa multidisciplinar para procurar ambiguidades ou omissões.Prototipagem: Construção de maquetes de interfaces de utilizador (UI/UX) de alta ou baixa fidelidade para recolher feedback imediato.Geração de Testes: Criação de planos de testes de aceitação com base nos requisitos; se um requisito não puder ser testado, ele deve ser rescrito.
5.5 Gerenciamento (Gestão) de RequisitosO processo de gerir as alterações inevitáveis que ocorrem nos requisitos durante e após o desenvolvimento do sistema.Componentes de Gestão:Rastreabilidade (Traceability): Capacidade de mapear a origem de um requisito (quem o pediu), a sua relação com outros requisitos, o componente de código onde foi implementado e o caso de teste que o valida.Controlo de Mudanças (Change Control): Processo formal que avalia o impacto de um pedido de alteração antes de este ser aceite pela equipa de desenvolvimento.Fluxo de Controlo de Mudanças:[Pedido de Alteração] ──► [Análise de Impacto e Custo] ──► [Avaliação do CCB (Comitê)] ──► [Se Aprovado
